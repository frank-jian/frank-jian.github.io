MySQL查询优化

发表于 2016-11-02   |   分类于 MYSQL   |  

前言

临近双11，达达App的流量突然猛增，日单量破百万单。这几天数据库出现几次报警，排查下来是某些SQL性能很差，导致慢查询。本节我们主要来学习，优化SQL。

查询原理

当我们向MySQL发送一个请求时，它到底做了什么？大概有一下几个步骤：

  1. 客户端发送一条查询给服务器
  2. 服务器先检查查询缓存，如果命中缓存则立刻返回存储在缓存中的结果，否则进入下一个阶段。
  3. 服务器进行SQL解析、预处理，再由优化器生成对应的执行计划
  4. MySQL根据优化器生成的执行计划，调用存储引用的API来执行查询。
  5. 将结果返回给客户端。

实战

1、查询执行计划

通过执行计划知道，type=index表示通过索引查询，预估得到2条记录rows=2;

2、不要使用count(非索引字段)

通过执行计划可知，使用count(非索引字段)使用全表扫描查询，而使用count(*)或count(索引字段)使用索引查询。

3、切分查询

场景： 有时候一个大查询我们需要分而治之，将大查询切分成小查询，每个查询功能完全一样，只完成一小部分，每次只返回一部分查询结果。

举个🌰

定期删除大量数据时，如果用一个大的语句一次性完成的话，则可能需要一次锁住很多数据、占满整个事务日志、耗尽系统资源、阻塞很多很小的但很重要的查询。将一个大的DELETE语句切分成多个小的查询可以尽可能小地影响MYSQL性能，同时还可以减少MYSQL复制的延时。

再举个线上环境的🌰

场景： 假设我需要一次查询1万个用户的信息；

大查询： SELECT * FROM ‘dada’ WHERE dada_id IN (1万个id);

小查询: SELECT * FROM ‘dada’ WHERE dada_id IN (1千个id); 查询10次；

结果： 使用大查询很可能出现慢查询，导致性能问题，而使用小查询，对服务器性能影响小很多。

4、分解关联查询

关联查询

分解查询

关联查询分解为多个查询的好处

  1. 缓存效率更高，如:上面查询中的tag=mysql已经被缓存了，那么应用就可以跳过第一个查询。再如，应用已经缓存post.id为123、456的内容，那么第三个查询的IN就可以少查询几个post.id。如果表结构变了，就不能使用查询缓存了。
  2. 查询分解后，执行单个查询可以减少锁的竞争。
  3. 应用层做分解后，可以更容易对数据库进行拆分，更容易做到高性能和可扩展。
  4. 查询效率会有所提升，使用IN代替关联查询，可让MYSQL按照ID顺序进行查询，这可能比随机关联更高效。
  5. 减少冗余记录的查询，在应用层做分解查询，意味着对于某条记录应用只需查询一次，而在数据库做关联查询，则需要重复访问一部分数据，可减少网路和内存的消耗。

JAVA之观察者模式

发表于 2016-10-29   |   分类于 JAVA   |  

前言

半年前写过一篇观察者模式,当我再次翻阅这篇文章时，我已看不懂，我当初写这篇文章的主线是什么？仅仅是将「Head First设计模式」书中的代码搬过来。这样没有经过思考的学习笔记，对知识的理解是很浅的，且忘记后再看原文不容易重新拾起。本文将重新梳理「JAVA观察者模式」。

观察者模式介绍

一个目标对象管理所有依赖于它的观察者对象，并且在它本身的状态改变时，主动发出通知。

优点

观察者和被观察者是抽象耦合的

缺点

  1. 如果一个目标对象有很多的观察者，通知所有观察者会花费很多时间
  2. 目标对象和观察者之间有循环依赖的话，会触发循环调用，导致系统崩溃。
  3. 观察者模式没有相应的机制让观察者知道目标对象是怎么发生变化的，仅仅只是知道观察目标发生了变化。

组成

  1. 抽象主题角色「Subject」:把所有观察者对象的引用保存在一个聚集里，每个主题都可以有任何数量的观察者。抽象主题提供一个接口，可以增加和删除观察者对象。
  2. 具体主题角色 「ConcreteSubject」：存入具体观察者对象，在具体主题的内部状态改变时，给所有登记过的观察者发出通知。
  3. 抽象观察者角色「Observer」：为所有的具体观察者定义一个接口，在得到主题的通知时，更新自己。
  4. 具体观察角色「ConsreteObserver」：具体观察者实现抽象观察者所要求的所有接口。

算法基础之插入排序

发表于 2016-10-29   |   分类于 算法   |  

前言

算法对一个工程师来说是极其重要的，初级工程师虽然在工作中，鲜有用到算法，但对算法的训练仍不能有丝毫的含糊，算法对工程师的思维训练也是有极大的好处的。从基础算法起步，逐步学习算法。本节主要介绍插入排序算法。

插入排序介绍

通过构建有序序列，对于未排序数据，在已排序序列中从后向前扫描，找到相应位置并插入。

举个实际生活中的例子

排序一手扑克牌，开始时，我们的左手为空并且桌子上的牌面向下，然后我们每次从桌子上拿走一张牌并将它插入左手中正确的位置。为了找到一张牌的正确位置，我们从右到左将它与已在手中的每张牌进行比较，拿在左手上的牌总是排序好的。

效率

时间复杂度

O(n^2)

空间复杂度

O(n),需要辅助空间O(1)

算法步骤

  1. 第一个元素开始，该元素可以认为已经被排序
  2. 取出下一个元素，在已经排序的元素序列中从后向前扫描
  3. 如果该元素「已排序」大于新元素，将该元素移动下一位置
  4. 重复步骤3，直到找到已排序的元素小于或者等于新元素的位置
  5. 将新元素插入到该位置后
  6. 重复步骤2~5

为何ArrayList能够动态扩容？

1. 定义变量

2. 初始化，实际元素个数为0

3.增加元素

ensureCapacityInternal 具体实现

简单叙述一下步骤：

  1. 初始化,元素个数(size=0)为0
  2. 此时elementData为空，故minCapacity=10
  3. 因minCapacity=10而elementData.length=0故第一次自动扩容。
  4. 当前元素的个数 * 1.5 = 0小于minCapacity=10,所以扩容的长度为10.
  5. 赋值elementData的长度为10。自此完成一次完整扩容。
  6. 再次增加元素，因minCapacity - elementData.length == 0，故不会再次扩容，直到元素的个数为11个，此时minCapacity=11而elementData.length=10，将再次触发扩容。

JAVA之简单工厂模式

发表于 2016-10-24   |   分类于 JAVA   |  

前言

上一节我们学习了模板方法，这一节让我们来学习简单工厂模式。

定义

工厂模式就是按照要求返回一个类型的对象。

目的

使对象的实例化与主程序代码解耦。

设计原则

  • 变量不要持有具体类的引用
  • 不要让类继承具体类，要继承抽象类或接口
  • 不要覆盖基类中已实现的方法

优点

  • 简单工厂模式实现了对责任的分割，提供了专门的工厂类创建对象
  • 引入配置文件，在不需要修改客户端代码的情况下更换和增加新的产品类，一定程度提高系统的灵活性。

缺点

  • 扩展困难，一旦添加新产品需要修改工厂逻辑
  • 产品类型较多时，可能造成工厂逻辑过于复杂，不利于系统扩展和维护
  • 工厂类集中了所有产品创建逻辑，一旦不能工作，整个系统将受影响。
  • 简单工厂模式会增加系统中类的个数，一定程度上增加了系统的复杂性和理解难度
  • 使用静态工厂方法，造成工厂角色无法形成基于继承的等级结构

应用场景A

我们与银行有业务来往，需要与银行方进行报文通信，不同的业务的报文不同，为此我们需要有一个工厂来简单创建不同的报文。

写一个Servlet处理客户端请求的时候，往往会给一个Servlet处理多个业务逻辑，为此我们需要有一个工厂类简单创建不同的Servlet。

Java之模板方法

发表于 2016-10-17   |   分类于 JAVA   |  

前言

模板方法之前写过一篇Java设计模式之模板方法模式,当回过头来再看这篇文章时，发现原来的文章过于累赘，无法让人快速理解，故重新梳理。

理论

什么是模板方法？

提取了类库中的公共行为，将公共行为放在父类中，通过其子类来实现不同的行为。

优点

  1. 将不变的行为搬到超类，去除了子类中的重复代码
  2. 子类实现算法的细节，有助于算法的扩展。
  3. 父类调用子类实现的操作，通过子类扩展新的行为，符合开闭原则。

缺点

  1. 每个不同的实现都需要定义一个子类，这会导致类的个数的增加，设计更加抽象。
